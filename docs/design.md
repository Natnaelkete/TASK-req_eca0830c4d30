# Agricultural Research Data & Results Operation Platform — Design

This document describes the architecture and key design decisions for the Agricultural Research Data and Results Operation Platform API. The system is a pure backend API built in Go, using Gin, GORM, MySQL, and Docker in a single-node offline environment.

## 1. Architecture Overview

### 1.1 High-Level Structure


The system is organized into layers:

- **API Layer (Handlers / Routers)**  
  - Implements HTTP endpoints with Gin.  
  - Performs request parsing, validation, and response formatting.  
  - Contains only thin logic and delegates to services.

- **Service Layer (Business Logic)**  
  - Implements domain-specific logic for:start
    - Monitoring domain
    - Analysis (indicator system and master data)
    - Communication (order conversations and notifications)
    - Results management
    - Evaluation tasks
    - Account system
    - Global audit and capacity monitoring
  - Orchestrates repositories and enforces business rules (state machines, rate limits, etc.).

- **Repository / Persistence Layer**  
  - Implements data access using GORM for MySQL.  
  - Encapsulates all queries for entities such as plots, devices, metrics, monitoring data, indicators, orders, conversations, results, tasks, users, and audit logs.  
  - Responsible for time-series partitioning and hot/cold archiving at the DB level.

- **Infrastructure Layer**  
  - Configuration (environment variables, app config).  
  - Logging.  
  - Background tasks (asynchronous queues, archiving, overdue task checks, capacity monitoring).  
  - Middleware (authentication, authorization, audit logging, rate limiting, sensitive word interception).

### 1.2 Domain Boundaries

The system is divided into domains based on the original prompt:

- **Monitoring Domain**
  - Monitoring Data Interface (batch write, query, trend computation).
  - Dashboard Configuration Interface (saved views, export).

- **Analysis Domain**
  - Indicator System Interface.
  - Master data for plots, devices, and metric dictionaries.
  - Indicator version management and auditability.
  - Analysis outputs for trends, funnels, and retention.

- **Communication Domain**
  - Order Conversation and Notification Interface.
  - Order-level conversations, read tracking, ticket transfer.
  - Template messages, sensitive word interception, and per-user rate limiting.

- **Results Management Domain**
  - Results Interface for papers, projects, and patents.
  - Configurable field rules (required fields, length limits, enumerations).
  - Strict status transitions and post-archive restrictions.

- **Evaluation Task Domain**
  - Task Orchestration Interface.
  - Task generation by objects and cycles, visibility windows, review node, and overdue handling.

- **Account and Security Domain**
  - Local username + password authentication.
  - Password and sensitive field handling (hashing/encryption/desensitization).
  - Role-based access where appropriate.

- **Audit and Capacity Domain**
  - Global audit logs of operations.
  - Capacity monitoring and threshold-based internal notifications.

## 2. Technology Stack

- **Language:** Go  
- **Web Framework:** Gin  
- **ORM:** GORM  
- **Database:** MySQL  
- **Containerisation:** Docker with docker-compose  
- **Environment:** Single-node offline deployment

No external cloud services, message queues, or external identity providers are used. All behaviour relies on the local MySQL database and the Go application.

## 3. Data Model Overview

### 3.1 Core Entities (Conceptual)

- **User**
  - Local account with username, password hash, role, and basic profile/security data.
- **Plot**
  - Represents a research plot (site). Includes identifiers and tags.
- **Device**
  - Represents a monitoring device, linked to a plot.
- **Metric / Indicator**
  - Represents measurable metrics and their definitions (indicator dictionary).
- **Metric Version**
  - Stores indicator versions, modifier, timestamp, and difference description.
- **Monitoring Data**
  - Time-series measurements for devices and plots, including `source_id`, `event_time`, `metric_code`, and value, with tags.
- **Dashboard Configuration**
  - Saved monitoring query configurations per user (filters, dimensions, time window).
- **Order**
  - Represents the unit on which conversations occur in the communication domain.
- **Conversation Message**
  - Messages within an order-level conversation, including sender, content, read status, and transfer details.
- **Result**
  - Research results such as papers, projects, or patents with configurable metadata and status.
- **Task**
  - Evaluation tasks generated by object and cycle, with visibility windows and review state.
- **Audit Log**
  - Records operator, timestamp, resource, and action for important operations.
- **Capacity Notification**
  - Records capacity-related notifications, such as disk usage threshold breaches.

### 3.2 Time-Series Partitioning and Archiving

- Monitoring data tables are **partitioned monthly** by timestamp.  
- **Hot data**:
  - Retained for 90 days.
  - Indexed to support sub-second queries for monitoring and dashboards.
- **Cold data**:
  - Archived into historical tables.
  - Retained for 3 years.
- A background process manages:
  - Moving data older than 90 days into historical storage.
  - Keeping current partitions optimized through indexes and pruning.

### 3.3 Idempotent Batch Writes

- Batch writes use an **asynchronous queue**:
  - Incoming monitoring records are added to an in-memory queue.
  - Background workers consume the queue and perform batched database writes.
- Idempotency:
  - A composite key `source_id`, `event_time`, and `metric_code` is used to avoid duplicates.
- Core indexes:
  - `(device_id, plot_id, metric_code, event_time)` to support time-series queries.

## 4. Key Design Decisions

### 4.1 Authentication & Authorization

- User authentication is implemented using local username + password validation.  
- Passwords are stored only as secure hashes.  
- Access control is applied at API endpoints and services where necessary to separate roles (administrator, researcher, reviewer, customer service).

### 4.2 Monitoring and Analytics Design

- Monitoring queries support:
  - Filtering by plot, device, metric, time window, and tag.
  - Metric aggregations for daily, weekly, and monthly trends.
  - Year-over-year and month-over-month comparisons where enough history exists.
- Dashboards store query parameters, not rendered graphs, so clients can re-construct charts.

### 4.3 Communication Safety (Rate Limits & Sensitive Words)

- Rate limiting for conversations:
  - A per-user limit of 20 messages per minute to prevent harassment.
- Sensitive word filtering:
  - Messages are checked for sensitive words before being stored or dispatched.
  - Intercepted messages are not delivered but are logged for audit.

### 4.4 Results Workflow

- Result types (papers, projects, patents) share a common state machine:
  - `draft → submitted → returned → approved → archived`
- Status transitions are controlled by service logic; only allowed transitions are permitted.
- After archived:
  - Only retrospective notes can be appended.
  - Invalidation requires a reason and is fully traceable through audit logs.

### 4.5 Task Orchestration

- Tasks are generated by object and cycle (e.g., daily, weekly, or other cycles represented consistently).
- Each task has:
  - A visibility window.
  - A submission point.
  - A review phase.
- Overdue tasks:
  - Automatically marked as delayed when exceeding the default 7-day overdue rule.

### 4.6 Audit and Capacity Monitoring

- Global audit logging:
  - Implemented as middleware or a service layer hook.
  - Captures who did what, when, and on which resource.
- Capacity monitoring:
  - Periodically checks resource usage (such as disk).
  - Triggers internal notifications when configurable thresholds are exceeded (e.g., >80% disk usage).

## 5. Deployment & Operations

- The application is packaged as a Docker image.
- A MySQL container is provided via docker-compose.
- `docker compose up` must:
  - Build and start all services.
  - Allow basic health checks from the README (e.g., health endpoint, basic API flows).
- No environment-specific dependencies or external services are required.

## 6. Testing Strategy

- **Unit tests** cover:
  - Business rules (state transitions, validation rules).
  - Utility functions and domain logic.
- **API tests** cover:
  - Endpoint behaviour with normal, boundary, and error conditions.
- All tests are run via `make test` (which runs `go test ./... -v -cover`) from the repository root.
- The testing setup is designed so that it can run inside Docker or against a Dockerized database.
